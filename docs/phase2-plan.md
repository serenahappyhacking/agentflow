# AgentFlow Phase 2 — 干掉 RD + AI 对话式云操作

> Phase 1（干掉运维）已完成，Phase 2 目标：用 LLM 替代外包研发 + 用 AI 对话替代填表

---

## 两个方向

### 方向 1：干掉 RD（LLM Vibe Coding）

用 LLM 自动写代码，替代外包研发。业务负责人描述需求 → AI 写码 → AI 审查 → 自动部署。

### 方向 2：AI 对话式云操作

用飞书机器人对话替代填审批表单。IT Director 反馈："AI 直接操作服务器更加简单，更加直接"。

---

## 方向 1：干掉 RD — 详细设计

### 核心流程

```
业务负责人在飞书描述需求（对话或上传PRD）
  → AgentFlow 解析需求
  → Coding Agent（LLM）clone 仓库 + 分析上下文 + 生成代码
  → Review Agent（另一个 LLM）审查代码（安全/质量/业务逻辑）
  → 审查通过 → 自动 push 到 Gitee
  → 云效自动构建部署（Phase 1 已有能力）
  → 飞书通知业务负责人：功能已上线
```

### 关键设计决策

| 决策点 | 方案 |
|--------|------|
| 需求输入 | 飞书对话 + PRD 文档上传 |
| 代码生成 | Coding Agent（Claude 或 DeepSeek） |
| 代码审查 | Review Agent（用不同模型交叉审查） |
| 人工参与 | 无（全自动），AI 审查作为安全网 |
| 部署 | 复用 Phase 1 云效流水线 |

### Coding Agent 能力

- 自动 clone Gitee 仓库，分析现有代码结构
- 根据需求生成新代码或修改已有代码
- 支持多语言：Java(Spring Boot) / Node.js(Express/Nest) / Python(FastAPI) / Vue / React
- 支持 tool use：LLM 自主决定读哪些文件、写什么代码、运行什么测试

### Review Agent 审查维度

- 安全漏洞（SQL注入、XSS、硬编码密码）
- 代码质量（命名规范、重复代码、错误处理）
- 业务逻辑（是否满足需求描述）
- 依赖安全（是否引入有漏洞的包）
- 审查不通过 → 反馈给 Coding Agent 修改（最多 3 轮）

### LLM 选型

| LLM | 代码能力 | 中文理解 | 成本 | 合规 | 建议角色 |
|-----|---------|---------|------|------|---------|
| Claude Opus/Sonnet | 最强 | 好 | 高 | 境外 | Coding Agent |
| DeepSeek V3 | 强 | 最好 | 低 | 国内 | Review Agent / 降本替代 |
| 通义千问 | 中 | 好 | 中 | 国内 | 备选 |

建议：先用 Claude 验证效果，后续切 DeepSeek 降本。两个 Agent 用不同模型形成交叉审查。

## 方向 2：AI 对话式云操作 — 详细设计

### 对比

| | Phase 1（当前） | Phase 2（新增） |
|---|---|---|
| 入口 | 飞书审批表单（填 7 个字段） | 飞书群 @机器人 一句话 |
| 示例 | 打开审批 → 选阿里云 → 选Redis → 填4G → 填项目名... | "帮我开一个4G Redis，给order-service用" |
| 耗时 | 3分钟填表 + 等审批 | 10秒发消息 + 确认 |

### 支持的对话指令

| 用户说 | 系统做 |
|--------|--------|
| "帮我开一个 4G Redis" | 开通云资源 |
| "查一下 order-service 用了哪些资源" | 查询资源台账 |
| "这个月花了多少钱" | 查询成本 |
| "把 order-test.example.com 换成 order.example.com" | 域名替换 |
| "帮我建一个 order-service 的部署流水线" | 创建流水线 |
| "order-service 的 Redis 什么时候到期" | 查询到期信息 |
| "给 order-service 加一个按日期筛选的功能" | **Vibe Coding（方向1）** |

### 安全分级

| 风险等级 | 操作类型 | 处理方式 |
|---------|---------|---------|
| 低 | 查询资源、查价格、查到期 | 直接执行，回复结果 |
| 中 | 开通测试环境资源 | 发确认卡片，用户点"确认"后执行 |
| 高 | 生产环境变更、大金额资源 | 自动发起飞书审批流（走 Phase 1 流程） |

### 交互示例

```
用户: @AgentFlow 帮我开一个 4G Redis，项目是 order-service

AgentFlow: 即将执行以下操作：
  ┌─────────────────────────┐
  │ 开通云资源               │
  │ 云厂商: 阿里云           │
  │ 类型: Redis 4G          │
  │ 项目: order-service      │
  │ 预估月费: ¥298           │
  │                         │
  │  [确认执行]  [取消]      │
  └─────────────────────────┘

用户: (点击 确认执行)

AgentFlow: ✅ Redis 已创建
  连接信息: r-bp1xxx.redis.rds.aliyuncs.com:6379
  密码: Ax7k...（已发送到你的单聊）
```

## 新增代码结构

```
agentflow/
├── agents/                          # ★ Phase 2 新增

│   ├── coding_agent.py              # 代码生成 Agent
│   ├── review_agent.py              # 代码审查 Agent
│   ├── context_builder.py           # 项目上下文（clone仓库 + 分析结构）
│   ├── intent_parser.py             # 自然语言 → 结构化操作意图
│   └── git_ops.py                   # Git 操作（clone/branch/commit/push）
├── api/
│   ├── lark_events.py               # Phase 1 已有（审批回调）
│   └── lark_bot.py                  # ★ 飞书机器人对话入口（两个方向共用）
├── llm/                             # ★ LLM 抽象层
│   ├── base.py                      # LLM 接口定义
│   ├── claude.py                    # Claude API
│   └── deepseek.py                  # DeepSeek API
└── (Phase 1 已有的 cloud/ lark/ models/ workflows/ 保持不变)
```

## 实施计划

### Phase 2.1（第1-2周）：AI 对话式云操作 — 投入小见效快

| 时间 | 交付 |
|------|------|
| 第1周 | 飞书机器人消息接收 + LLM 意图识别 + LLM 抽象层 |
| 第2周 | 对接 Phase 1 workflows + 确认卡片交互 + 查询指令 + 端到端测试 |
| **交付物** | **飞书群 @机器人 说话就能操作云资源** |

### Phase 2.2（第3-5周）：LLM Vibe Coding — 核心能力

| 时间 | 交付 |
|------|------|
| 第3周 | DeepSeek 接入 + 项目上下文构建 + 基础代码生成 |
| 第4周 | AI 代码审查 + Git 自动操作 + 全链路串联 |
| 第5周 | 任务编排 + 飞书对话集成 + 效果对比测试 |
| **交付物** | **飞书描述需求 → AI 写码 → AI 审查 → 自动部署上线** |

### Phase 2.3（第6周）：优化

- 模型效果对比报告（Claude vs DeepSeek vs 通义千问）
- 成本优化（简单任务用便宜模型，复杂任务用强模型）
- 生成代码质量统计
- 安全加固

---

## 风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| LLM 生成代码有 bug | Review Agent 交叉审查 + 测试环境先验证 |
| LLM 不理解业务上下文 | context_builder 读取仓库现有代码 |
| 对话误解执行错误操作 | 确认卡片 + 高风险走审批 |
| 境外 LLM 合规问题 | 可切 DeepSeek/通义千问 |
